# WebLFP 使用说明

这份说明面向主要关心实验数据和结果解释、但不熟悉软件配置的研究人员。WebLFP 是本地运行的 LFP 表征工具：数据留在本机，浏览器只是用来操作本机服务。

## 1. WebLFP 做什么

WebLFP 读取本地 LFP 记录，按预训练模型要求完成预处理和切窗，然后生成每个时间窗的 256 维 LFP feature。该 feature 来自原项目 `spike_count/clip` 统一权重，可直接接入随权重保存的窄波/非窄波 presence 与 count 分类头。

需要注意：

- 推理时只需要 LFP，不需要 Spike。
- WebLFP 不重建 Spike 波形。
- 输出不是自动诊断结论，也不能直接外推为神经元类型结论。
- 当前模型的证据主要来自报告所述小鼠海马记录和相关任务。

## 2. 使用前准备

请先确认三件事：

1. 记录文件在当前电脑或可访问的存储设备上。
2. 你知道记录格式，或至少知道它来自哪种采集系统。
3. 对于普通数组文件或原始二进制文件，你知道采样率；原始二进制还需要知道数据类型、通道数和数据排列方式。

如果不确定格式，可以先选择“自动”，再根据错误提示补充信息。

## 3. 打开应用

启动 WebLFP 后，用浏览器打开启动窗口提示的地址。页面顶部会显示模型权重状态；如果显示权重不可用，请先确认 Git LFS 已下载内置模型文件。

项目初始化不会安装 PyTorch。首次打开时，应用会要求先进入设置页检查计算环境并安装合适的版本：NVIDIA GPU 使用驱动支持的 CUDA 13.0 或更高版本，受支持的 Linux AMD GPU 使用 ROCm，没有兼容 GPU 时使用 CPU。安装并重新检测后点击“使用当前环境”，才能返回工作区。

应用主要有四个页面：

- 工作区：导入记录、预览信号、运行推理和导出结果。
- 历史：重新查看或删除本机已完成的 runs。删除会同时移除结果配置和数组文件，且无法恢复。
- 设置：检查计算环境、安装合适的 PyTorch 版本、检查 GitHub 更新，以及切换中英文和深浅色界面。
- 说明：查看这份面向使用者的操作说明。

界面首次打开默认使用英文，可在设置页切换为中文。语言偏好会保存在当前浏览器中。

## 4. 导入 LFP 记录

在工作区右侧的“本地路径”中可以手动输入文件或目录位置，也可以点击“选择文件”。选择文件不会上传数据，只是把本机文件位置填入输入框。

常见格式包括：

- NumPy 数组文件。
- MATLAB 数组文件。
- 原始二进制文件。
- 常见电生理采集系统产生的记录目录或文件。
- NWB 文件。

目录型记录目前可能仍需要手动输入目录位置。原始二进制文件必须填写数据类型、通道数、采样率和布局；这些信息不能由软件可靠猜测。

## 5. 检查元数据

点击“检查记录”后，WebLFP 会先读取元数据，不会立即载入完整数据。请确认：

- 采样率是否正确。
- 通道数量是否符合预期。
- 记录时长是否合理。
- 数据类型和读取格式是否符合你的记录。

如果这些信息明显不对，请先修正格式或参数，再继续。

## 6. 选择时间和通道

建议先选择较短的一段时间做预览，单次预览最长支持 300 秒。右侧可以设置：

- 开始时间和结束时间，单位为秒。
- 通道 ID：在搜索框中输入部分名称，从记录元数据返回的候选项中多选，无需事先知道完整 ID。
- 推理设备，可选择自动、CPU 或 CUDA。

通道过多会增加内存和显存占用。初次检查时，可以先使用少量代表性通道。

## 7. 预览处理结果

运行推理前，请先点击“预览处理结果”。页面会显示原始波形和处理后波形。读取完成后，选定的原始片段会保存在内存中；只要数据源、数据流、时间范围和通道没有变化，后续推理会复用该片段，不会再次读取记录文件。

请重点检查：

- 波形是否看起来像预期的 LFP。
- 是否存在全零、全常数或明显坏道。
- 时间范围是否覆盖你关心的实验阶段。
- 处理后波形是否没有明显异常放大。

如果预览结果不合理，请不要直接运行推理；先调整通道、时间范围或读取参数。

## 8. 运行 LFP-only 推理

预览确认后，点击“运行 LFP-only 推理”。WebLFP 会优先复用内存中的目标片段，并按照模型配置进行：

1. 读取选定时间和通道。
2. 必要时重采样到模型需要的采样率。
3. 切分为固定长度的时间窗。
4. 对每个通道进行 robust z-score 归一化。
5. 使用 LFP 分支生成 256 维 feature。

整个过程不读取 Spike，也不执行 Spike 排序。

## 9. 查看结果

结果页会显示：

- 有效时间窗数量。
- feature 维度。
- 实际运行设备。
- 特征范数检查。
- 可用鼠标旋转、缩放和平移的 UMAP 三维表征空间。
- 相邻时间窗的余弦相似度。

这些图用于观察表征随时间变化，不应单独当作生理标签或诊断结论。

## 10. 窄波 / 非窄波活动解码

LFP feature 生成后，可以点击“运行 presence 与 count”，执行原项目中的参考下游任务。无需选择额外的分类文件，应用会使用同一份已校验 SHA-256 的内置权重。

系统会直接读取当前 run 已保存的 256 维 feature，并为每个 0.2 秒窗口分别输出：

- 窄波 presence 概率与是否出现。
- 非窄波 presence 概率与是否出现。
- 窄波预测 count。
- 非窄波预测 count。

presence 概率达到 0.5 时标记为出现。count 图显示模型的连续预测值；原项目进行整数计数评估时，会把它四舍五入并限制为不小于 0。

当前 WebLFP 使用的是原项目 `spike_count/clip` 的统一 inference-only 权重：其中包含 LFP feature_extractor 和 SpikeCountPresenceHead。下游解码直接使用结果页生成并保存的 256 维 feature，不会重新读取原始记录，也不会使用未训练的线性适配层。

该输出是窗口级窄波/非窄波活动代理，不是单神经元真实细胞类型。它不能替代组织学、转录组、膜片钳或其他独立验证。页面中的论文参考指标来自 KA 慢性癫痫小鼠海马数据，不能直接外推到其他数据分布。

## 11. 导出结果

推理完成后可以导出：

- embeddings 文件：包含每个时间窗的 256 维 LFP feature。
- run 配置文件：记录读取参数、通道、时间范围、模型信息和处理流程。

请保存 run 配置文件。它能帮助你之后复现实验条件，也方便与同事核对结果。

## 12. 设置页面

设置页面用于检查运行环境：

- 当前 PyTorch 版本。
- CPU、CUDA 或 ROCm 后端。
- GPU、显存、CUDA 和 cuDNN 状态。
- 系统内存和内存有效传输率。
- CUDA BF16 简短性能测试。

初始环境不包含 PyTorch。请按 GPU 配置选择并安装 PyTorch：NVIDIA 用户应选择页面显示为兼容的 CUDA 13.0 或更高构建；AMD 用户只能在受支持的 Linux 系统上选择 ROCm；CPU 模式也是有效选择。安装完成并确认检测结果后，点击“使用当前环境”。

如果显示“机器性能过差”，说明当前电脑可能仍能运行小片段预览，但不适合处理较大数据或长时间范围。可以先缩短时间范围、减少通道数量，或改用性能更好的工作站。

更换 PyTorch 版本时，WebLFP 会停止本机服务，安装完成后自动重启。安装窗口显示日志；关闭日志窗口不会中断后台安装。

设置页还可以切换 English / 中文，以及跟随系统、深色或浅色外观。首次打开默认使用英文。

## 13. 常见问题

### 为什么选择文件后没有上传进度？

WebLFP 是本地应用。选择文件只是告诉本机服务要读取哪个记录，并不会上传到外部服务器。

### 为什么必须填写采样率？

对数组文件和原始二进制文件，采样率通常不在文件中。采样率错误会导致时间窗划分和重采样错误，因此必须填写真实值。

### 为什么不能直接分析很长一段数据？

长时间范围会产生大量窗口，占用更多内存、显存和时间。建议先用短片段确认读取和预处理正确，再扩大范围。

### 结果能代替 Spike 吗？

只能在经过验证的下游任务中“部分替代”。它不是 Spike 波形，也不是完整神经活动的直接重建。

### 出现错误怎么办？

先看错误提示涉及哪个阶段：读取、预览、推理、模型或设备。通常可以从以下方面排查：

- 文件是否可访问。
- 格式是否选对。
- 采样率、通道数、数据类型是否正确。
- 选择的时间范围是否在记录时长内。
- CUDA 不可用时是否可以切换到 CPU 先验证小片段。
